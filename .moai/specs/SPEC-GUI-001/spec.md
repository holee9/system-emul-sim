## 1. Problem Statement

The X-ray Detector Panel System has **working 4-layer simulators** (Panel→FPGA→MCU→Host)
verified by IT-11 bit-exact tests, but the **GUI is a standalone demo facade** that:
- Generates fake 256×256 synthetic frames (not from PanelSimulator)
- Never instantiates PipelineDetectorClient
- Has non-functional Start/Stop/Reset buttons
- Shows hardcoded zeros in Pipeline Status tab
- Runs fake scenario animations (always "Passed=true")

**Goal of this SPEC**: Connect the existing, verified 4-layer pipeline to the GUI
in incremental, user-verifiable milestones.

## 2. User Stories (Acceptance Test Basis)

Each story has a clear User Acceptance Test (UAT) that defines "Done."

### US-001: Real Frame Display (MVP-1)
> As a user, I want to launch the GUI and see frames generated by the actual
> 4-layer simulator pipeline, not synthetic test data.

**UAT**: User launches GUI → Tab 2 (Frame Preview) shows frames with panel-physics-based
pixels. When user selects Counter pattern, pixel values increment monotonically.
When user selects Gaussian noise pattern, pixels vary per Gaussian distribution.

---

### US-002: Panel Parameter Control (MVP-1)
> As a user, I want to change panel resolution and see the frame dimensions change.

**UAT**: User changes PanelRows from 256 to 512 → Frame Preview updates to show
512-pixel-tall frame. Status bar shows "512 × 512" dimensions.

---

### US-003: Pipeline Status (MVP-2)
> As a user, I want to see real-time statistics for each pipeline layer while
> the simulator is running.

**UAT**: User starts pipeline → Tab 5 (Pipeline Status) shows live frame counts,
FPS, packet counts. Values increment as frames are processed. Stopping pipeline
freezes the counters.

---

### US-004: Noise and Defect Control (MVP-2)
> As a user, I want to adjust noise level and pixel defect rate and see the
> effect on simulated frames.

**UAT**: User increases noise sigma from 150 to 500 → Frame Preview shows
noticeably noisier image. User sets defect rate to 1% → Isolated dark/bright
pixels appear scattered in the frame.

---

### US-005: Scenario Execution (MVP-3)
> As a user, I want to run a predefined test scenario (e.g., IT-01 full pipeline)
> and see an actual PASS or FAIL result with details.

**UAT**: User selects "IT-01: Full Pipeline (256×256)" → Clicks Run → Progress bar
advances → Final result shows "PASSED: 100 frames, 0 dropped, bit-exact verified"
OR actual failure reason. Result is NOT always "Passed=true".

---

### US-006: Network Impairment Simulation (MVP-3)
> As a user, I want to simulate packet loss and see its effect on frame reception.

**UAT**: User sets PacketLossRate to 5% → Runs 100 frames → Pipeline Status tab
shows ~5% dropped packets. Frame dropout metric matches configured loss rate
within ±2%.

---

### US-007: PDF Parameter Extraction (MVP-4)
> As a user, I want to load a vendor panel specification PDF and have the
> key parameters automatically populated in the simulator configuration.

**UAT**: User loads a panel PDF → Tab 3 (Parameter Extraction) shows extracted
parameters including at minimum: Rows, Cols, BitDepth, PixelPitch, MaxFPS, NoiseFloor.
User clicks "Apply" → Simulator Control tab values update.
User starts simulation → New config is used (different from defaults).

---

### US-008: Frame Export (MVP-4)
> As a user, I want to save a captured frame as a 16-bit TIFF or RAW file.

**UAT**: User clicks "Save Current Frame" → File dialog opens → User selects
output path → File is saved → User can open file in ImageJ and verify
it matches the displayed frame (same dimensions, 16-bit depth, similar pixel values).

## 3. Technical Architecture

### 3.1 The Connection Gap (Root Cause)

```
Current (broken):
  App.xaml.cs
    ↓ creates
  SimulatedDetectorClient (generates 256×256 fake synthetic data)
    ↓ feeds
  GUI (shows fake data)

Required (fixed):
  App.xaml.cs
    ↓ creates
  SimulatorPipeline(PanelConfig, FpgaConfig, McuConfig, NetworkConfig)
    ↓ wrapped by
  PipelineDetectorClient (already exists at Services/PipelineDetectorClient.cs)
    ↓ feeds
  GUI (shows real 4-layer pipeline data)
```

## 4. MVP Milestones

### MVP-1: Real Pipeline Connection
**Sprint**: 3-5 days
**User Stories**: US-001, US-002

#### Requirements (EARS format)

**REQ-GUI-001** [MANDATORY]
When the application starts, the system shall initialize a `SimulatorPipeline`
with default `PanelConfig` (256×256, Counter pattern) and use `PipelineDetectorClient`
as the active detector client.

**REQ-GUI-002** [MANDATORY]
When the user selects a test pattern from the Simulator Control tab, the system
shall update `PanelConfig.Pattern` and restart the pipeline within 500ms.

**REQ-GUI-003** [MANDATORY]
When the pipeline generates a frame, the Frame Preview tab shall display the
actual pixel data from `SimulatorPipeline.ProcessFrame()`, not synthetic data.

**REQ-GUI-004** [MANDATORY]
When the user changes PanelRows or PanelCols and applies the change, the system
shall reconfigure the pipeline with the new resolution. Frame Preview dimensions
shall reflect the new size.

#### Acceptance Criteria

| ID | Criterion | Verification |
|----|-----------|-------------|
| AC-GUI-001 | App starts without exception using PipelineDetectorClient | Launch app, no crash |
| AC-GUI-002 | Counter pattern shows monotonically incrementing pixels | Zoom in on Frame Preview, inspect pixel values |
| AC-GUI-003 | Changing resolution from 256→512 changes displayed frame size | Observe Frame Preview height doubles |
| AC-GUI-004 | No regression in existing UI functionality (tabs load, window/level works) | Full UI walkthrough |

#### UAT Procedure
1. Launch GUI.Application.exe
2. Tab 2: Verify frame is displaying and updating (not frozen)
3. Tab 4 → Set Pattern to "Counter" → Click Apply
4. Tab 2: Zoom in → Verify pixel gradient from dark to bright
5. Tab 4 → Change Rows=512, Cols=512 → Click Apply
6. Tab 2: Verify frame is taller (512px vs 256px)
7. Tab 4 → Set Pattern to "Flat Field" → Click Apply
8. Tab 2: Verify uniform gray frame

**DONE when**: All 4 UAT steps complete without error.

### MVP-2: Live Pipeline Statistics + Parameter Control
**Sprint**: 3-5 days
**User Stories**: US-003, US-004
**Depends on**: MVP-1

#### Requirements

**REQ-GUI-005** [MANDATORY]
While the pipeline is running, the Pipeline Status tab shall display at minimum:
frames processed (count), frames per second (float, 1 decimal), and last frame
latency (milliseconds).

**REQ-GUI-006** [MANDATORY]
The polling interval for pipeline statistics shall be configurable between
100ms and 2000ms. Default: 500ms.

**REQ-GUI-007** [MANDATORY]
When the user adjusts the noise sigma slider, the system shall apply the new
noise configuration to PanelSimulator within one polling cycle.

**REQ-GUI-008** [MANDATORY]
When the user sets defect rate > 0%, the Frame Preview shall show pixel defects
(dead = 0, bright = 65535) corresponding to the configured rate.

#### Acceptance Criteria

| ID | Criterion | Verification |
|----|-----------|-------------|
| AC-GUI-005 | Pipeline Status tab shows non-zero frame count after 5 seconds | Observe tab, count > 0 |
| AC-GUI-006 | FPS display updates at least every 1 second | Watch for value change |
| AC-GUI-007 | Noise sigma change visible in frame within 1 second | Increase sigma, observe noisier frame |
| AC-GUI-008 | Defect rate 1% produces ~1% dark/bright pixels | Set 1%, observe scattered defects |

---

### MVP-3: Real Scenario Execution
**Sprint**: 5-7 days
**User Stories**: US-005, US-006
**Depends on**: MVP-1, MVP-2

#### Requirements

**REQ-GUI-009** [MANDATORY]
The Scenario Runner tab shall list at minimum the following real scenarios:
IT-01 (Full Pipeline, 10 frames), IT-08 (Packet Loss 5%), IT-09 (Stress, 100 frames).

**REQ-GUI-010** [MANDATORY]
When the user runs a scenario, the system shall execute it using the actual
`SimulatorPipeline` and `IntegrationTests` infrastructure, not a mock animation.

**REQ-GUI-011** [MANDATORY]
Scenario results shall report: frames requested, frames completed, frames failed,
execution time, and PASS/FAIL verdict with failure reason if applicable.

**REQ-GUI-012** [MANDATORY]
A scenario shall FAIL and report the actual reason when: frame count mismatch,
bit-exact verification failure, or exception during execution.

**REQ-GUI-013** [MANDATORY]
When PacketLossRate is set to N%, the scenario execution report shall show
dropped packet count within ±2% of N × total_packets.

#### Acceptance Criteria

| ID | Criterion | Verification |
|----|-----------|-------------|
| AC-GUI-009 | IT-01 runs and shows "10/10 frames, bit-exact PASSED" | Execute IT-01 |
| AC-GUI-010 | IT-01 with injected error shows actual FAILED result | Modify pipeline to inject error, run IT-01 |
| AC-GUI-011 | IT-08 at 5% loss shows ~5% packet drop in results | Set 5% loss, run IT-08, verify count |
| AC-GUI-012 | Scenario cannot be started while another is running | Click Run twice rapidly |

---

### MVP-4: PDF Parameter Extraction + Frame Export
**Sprint**: 5-7 days
**User Stories**: US-007, US-008
**Depends on**: MVP-2

#### Requirements

**REQ-GUI-014** [MANDATORY]
When the user loads a PDF, the system shall attempt to extract at minimum these
parameters: Rows, Cols, BitDepth, PixelPitch, MaxFPS, NominalNoiseSigma.

**REQ-GUI-015** [MANDATORY]
Each extracted parameter shall display its source (page number, extracted text
snippet) so the user can validate the extraction.

**REQ-GUI-016** [MANDATORY]
Parameters not found in the PDF shall be marked "Not found — using default"
rather than silently using a default value.

**REQ-GUI-017** [MANDATORY]
When the user clicks "Apply to Simulator", all extracted parameters shall be
written to the active pipeline configuration and take effect within one frame.

**REQ-GUI-018** [MANDATORY]
The user shall be able to save the current frame as a 16-bit grayscale TIFF or
RAW binary file via a "Save Frame" button in the Frame Preview tab.

**REQ-GUI-019** [MANDATORY]
Saved TIFF files shall be readable by ImageJ/Fiji without plugin configuration
and display the same pixel values as shown in Frame Preview.

#### Acceptance Criteria

| ID | Criterion | Verification |
|----|-----------|-------------|
| AC-GUI-013 | PDF load shows extracted parameters with source evidence | Load sample PDF, inspect results |
| AC-GUI-014 | "Not found" shown for missing params | Load PDF missing BitDepth, verify label |
| AC-GUI-015 | Apply causes simulation to use new resolution | Apply 1024×1024 from PDF, verify frame size |
| AC-GUI-016 | Save Frame → TIFF readable in ImageJ, 16-bit depth | Save, open in ImageJ, check Image > Info |

## 7. Test Strategy

### 7.1 Unit Tests (TDD — Red-Green-Refactor)

For each changed component, write test BEFORE implementation:
- `PipelineDetectorClientTests`: Mock SimulatorPipeline, verify frame delivery
- `MainViewModelTests`: Verify UpdateConfig() called on param change
- `PipelineStatusViewModelTests`: Verify stats polling calls GetStatistics()
- `ScenarioRunnerTests`: Verify actual test execution (not mock animation)
- `ParameterExtractorServiceTests`: Verify all 6+ params extracted and mapped

### 7.2 Integration Tests (Manual — UAT procedures above)

Each MVP milestone has explicit UAT steps that constitute integration testing.
UAT steps are the SOLE arbiter of "Done" — test count is secondary.

### 7.3 Regression Tests

Before each MVP release, run:
```powershell
dotnet test tools/GUI.Application/ --filter "FullyQualifiedName~Unit" --logger "console;verbosity=normal"
```

All 83 existing unit tests must continue to pass.

## 8. Definition of Done

A feature is DONE when:
1. User can perform the action described in the UAT without technical knowledge
2. Unit tests for the changed code pass
3. No regression in existing unit tests
4. No unhandled exceptions in the GUI during UAT

A feature is NOT DONE when:
- Tests pass but the user cannot actually perform the action
- Output is hardcoded or mocked
- Works only with specific config, not general use

- Network streaming over actual 10GbE

- Multi-panel configurations
- 3072×3072 resolution (requires real HW for throughput validation)
